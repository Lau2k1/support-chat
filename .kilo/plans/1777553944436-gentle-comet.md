# Баг: client_id UUID vs INTEGER в БД

## Проблема

В `ws/handler.ts:51` генерируется `crypto.randomUUID()` для `client_id`, но существующая колонка `chats.client_id` в БД имеет тип `INTEGER`. PostgreSQL отказывается вставлять UUID-строку в integer-колонку.

Ошибка: `invalid input syntax for type integer: "3c7844ee-dabf-4800-b07a-b6c7ff40656f"`

## Решение

Два варианта:

**Вариант A**: Изменить тип колонки в БД через ALTER TABLE (требует миграцию, может сломать существующие данные)

**Вариант B**: Использовать integer client_id — генерировать через `nextval` последовательности или просто использовать автоинкремент. Это не требует изменения схемы БД.

Выбираю **Вариант B** — минимум риска, не ломает существующую БД.

### Изменения

1. **`backend/src/ws/handler.ts`** — заменить `crypto.randomUUID()` на генерацию числового ID:
   - Убрать `import crypto from 'crypto'`
   - В `init_chat`: получить следующий client_id через `SELECT nextval('client_id_seq')` или `SELECT COALESCE(MAX(client_id), 0) + 1 FROM chats`
   - Проще: `INSERT INTO chats (client_id, status, updated_at) VALUES (DEFAULT, 'open', CURRENT_TIMESTAMP)` — если client_id имеет DEFAULT nextval

   Самый простой подход: использовать `SELECT COALESCE(MAX(client_id), 0) + 1 FROM chats` для получения нового client_id.

2. **`backend/src/db/schema/init.sql`** — вернуть `client_id` в INTEGER с DEFAULT:
   ```sql
   client_id INTEGER DEFAULT nextval('client_id_seq')
   ```
   Или просто `client_id INTEGER NOT NULL DEFAULT 1` и генерировать на стороне приложения.

3. Убрать неиспользуемый `import crypto` из handler.ts
